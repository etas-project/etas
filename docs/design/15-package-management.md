# Package Management

## 1. Goal

Package management is part of the Etas language specification at the project
and tooling boundary. It should not add source-language keywords. Source files
use logical `module` and `import` paths; package manifests, lockfiles, package
metadata, and registry configuration decide where those logical paths come from.

This separation keeps source code stable across local packages, registries,
GitHub dependencies, private registries, generated modules, and compiler-known
standard-library modules.

## 2. Source Language Boundary

Etas source code imports logical module paths:

```etas
module app.main;

import std.io.*;
import company.agents.Writer;
import web.search.SearchClient;
```

The source language does not support raw dependency locations:

```etas
// Not source syntax.
import "https://github.com/org/package";
import "../local/file.es";
```

Dependency sources, versions, aliases, registry URLs, Git revisions, and local
paths belong in `etas.toml` and `etas.lock`, not in `.es` source files.

## 3. Package Layout

The default package layout is:

```text
my-agent-app/
    etas.toml
    etas.lock
    src/
        app/main.es
        app/pipeline.es
    tests/
    examples/
```

If `src/app/main.es` declares:

```etas
module app.main;
```

then the default resolver maps the logical module path to one of:

```text
src/app/main.es
src/app/main/mod.es
```

If both files exist and both define `module app.main`, the package is ambiguous
and must be rejected.

## 4. Package Manifest

An Etas package is described by `etas.toml` at the package root.

```toml
[package]
name = "research-assistant"
version = "0.1.0"
edition = "2026"
license = "MIT"
description = "A multi-agent research assistant"
repository = "https://github.com/org/research-assistant"

[source]
root = "src"

[dependencies]
std = { version = "0.1" }
company_agents = { package = "company-agents", version = "1.2", import = "company.agents" }
web_search = { package = "web-search", version = "0.4", import = "web.search" }
local_prompts = { path = "../local-prompts", import = "prompts" }
github_tools = { git = "https://github.com/org/etas-tools.git", tag = "v0.3.0", import = "tools" }

[[bin]]
name = "research-assistant"
module = "app.main"
flow = "main"
```

Manifest fields:

| Field | Meaning |
|---|---|
| `[package].name` | Registry package name. |
| `[package].version` | Package version using SemVer. |
| `[package].edition` | Language edition used for parsing and semantic rules. |
| `[source].root` | Source root for the current package. Defaults to `src`. |
| `[dependencies]` | Package dependencies from registry, local path, Git, or built-in source. |
| `[[bin]]` | Runnable entry points. Each entry names a module and a flow. |

A dependency key is a local manifest name. It does not need to match the
registry package name or import root:

```toml
[dependencies]
company_agents = { package = "company-agents", version = "1.2", import = "company.agents" }
```

This lets source code import:

```etas
import company.agents.Writer;
```

If `import` is omitted in a dependency entry, the dependency key is used as the
default import root.

## 5. Dependency Sources

Etas should support these dependency source forms:

```toml
[dependencies]
std = { version = "0.1" }
local = { path = "../local-package", import = "local" }
tools = { git = "https://github.com/org/tools.git", rev = "abc123", import = "tools" }
tools_tag = { git = "https://github.com/org/tools.git", tag = "v0.3.0", import = "tools" }
tools_branch = { git = "https://github.com/org/tools.git", branch = "main", import = "tools" }
private = { registry = "company", package = "agent-tools", version = "2.0", import = "company.tools" }
```

Rules:

- registry dependencies resolve through a configured registry index;
- path dependencies resolve relative to the package manifest;
- Git dependencies must be locked to an exact revision in `etas.lock`;
- built-in packages such as `std` may be compiler-known, but they still expose
  normal package metadata;
- source code only sees logical import roots, not dependency source forms.

## 6. Version Requirements

Etas package versions use SemVer. The package manager should support:

```toml
[dependencies]
std = { version = "0.1" }          # compatible with 0.1.x
foo = { version = "^1.2.0" }       # >=1.2.0, <2.0.0
bar = { version = "=1.2.3" }       # exactly 1.2.3
baz = { version = ">=1.2,<1.5" }
```

The exact selected versions are recorded in `etas.lock`.

## 7. Lockfile

`etas.lock` is generated by tooling. Application packages should commit it so
builds are reproducible. Library packages may commit it, but public dependency
constraints still come from `etas.toml`.

Example:

```toml
version = 1

[[package]]
name = "std"
version = "0.1.4"
import_root = "std"
source = "builtin"
checksum = "builtin:std"

[[package]]
name = "company-agents"
version = "1.2.3"
import_root = "company.agents"
source = "registry+https://packages.etas.dev"
checksum = "blake3:..."
dependencies = ["std"]

[[package]]
name = "web-search"
version = "0.4.1"
import_root = "web.search"
source = "git+https://github.com/org/web-search.git?rev=abc123"
checksum = "blake3:..."
dependencies = ["std"]
```

Each lockfile entry records the resolved `import_root` that source imports use
for that package in this package graph. The value comes from the dependency
entry's `import` field, or from the dependency key when `import` is omitted.

The frontend cache key must include the resolved lockfile entries and package
metadata fingerprints for all dependencies that affect type checking, effect
checking, policy checking, generated tool schemas, and public API resolution.

## 8. Package Metadata

Published packages should include generated public API metadata. The metadata
lets the compiler type-check downstream packages without loading all dependency
source code.

The authoritative metadata artifact is a sectioned binary file:

```text
.etas/package.etasmeta
```

The artifact header records the package identity, compiler schema version,
source payload hash, manifest hash, dependency lock hash, and target that
created it. Section payloads carry exports, type contracts, effect contracts,
action contracts, tool contracts, policy contracts, std contracts, package graph,
and public symbols. JSON output is only a debug dump format; it is not a build
input.

```text
EtasMetadataArtifact
  header
  section table
  compressed protobuf sections
```

Public metadata should include:

- public modules and exported item names;
- public type, enum, spec, flow, agent, tool, effect, action, and protocol
  signatures;
- generated effect/action summaries for exported flows, agents, and tools;
- tool schemas for model-callable public tools;
- trace-spec summaries that affect exported APIs;
- re-export expansion for wildcard public imports;
- fingerprints for incremental compilation and LSP invalidation.

## 9. Import Resolution

Resolving:

```etas
import company.agents.Writer;
```

uses the package graph:

```text
1. Search current package modules.
2. Search dependency import roots from etas.toml.
3. Choose the longest matching import prefix.
4. Reject ambiguous matches.
5. Resolve the result to PackageId + ModuleId + ItemId.
```

For:

```toml
[dependencies]
company_agents = { package = "company-agents", version = "1.2", import = "company.agents" }
```

the import lowers to a stable identity:

```text
PackageId(company-agents@1.2.3) + ModuleId(company.agents) + ItemId(Writer)
```

This identity, not the physical source path, is what the type checker, effect
checker, policy checker, LSP, FIR, AIR, and runtime-facing metadata should use.

## 10. Workspace

A multi-package repository may define an `etas.workspace.toml`:

```toml
[workspace]
members = [
    "packages/core",
    "packages/web",
    "examples/*"
]

[workspace.dependencies]
std = { version = "0.1" }
```

Workspace configuration is tooling-level metadata. It does not change source
language semantics. A workspace dependency gives member packages a shared
dependency constraint when a member chooses to inherit it.

## 11. Registry And User Configuration

User-level package manager configuration belongs outside project source, for
example:

```text
~/.etas/config.toml
```

Example:

```toml
[registries.default]
index = "https://packages.etas.dev/index"

[registries.company]
index = "https://packages.company.com/etas/index"

[cache]
root = "~/.etas/cache"
```

Authentication tokens must not be written into `etas.toml` or `etas.lock`.
They should be loaded from environment variables, the system keychain, hosted
runtime configuration, or Etas secret references.

## 12. Tool Bindings

A package may expose a bodyless tool signature:

```etas
public tool search(query: SearchQuery) -> SearchResults ![AcademicSearch.search];
```

The signature is a typed source-level contract. A runnable application must still
resolve it to a concrete host binding through package metadata or runtime
configuration:

```toml
[bindings.tools]
"web.search.search" = { kind = "mcp", server = "browser", tool = "search" }
```

Rules:

- a tool with an Etas body does not need an external binding;
- a bodyless tool in a dependency may be type-checked from package metadata;
- a final runnable package must resolve all bodyless tool signatures reachable
  from its entry points;
- tool bindings must declare the effect/action row they may perform, and that
  row must be included in the source-level tool signature;
- binding configuration is not source syntax.

## 13. Project Compilation Model

Project-level compilation starts from a package root, not from an isolated source
file. A source file can still be parsed or checked alone for editor feedback, but
a runnable or publishable package is compiled through the package model.

The project compilation input is:

```text
etas.toml
etas.lock?
[source].root/**/*.es
dependency package metadata or dependency source
runtime/tool binding metadata when compiling a runnable target
```

The project compilation output is one or more of:

```text
checked package
public package metadata
runnable entry artifact
diagnostics
LSP/index metadata
```

### 13.1 Compilation Steps

The compiler should process a package in this order:

1. Discover the package root by locating `etas.toml`.
2. Parse `etas.toml` and determine package name, version, edition, source root,
   dependencies, and declared entry points.
3. Resolve dependencies from `etas.lock` when present. If the lockfile is absent
   or stale, package management tooling resolves dependency constraints and
   produces a new lockfile before normal compilation.
4. Build the package graph from the current package plus resolved dependencies.
5. Collect source files from `[source].root`, normally `src`.
6. Parse source files into ASTs.
7. Validate module declarations against the resolved logical module paths.
8. Build the module index for the current package.
9. Load dependency public API metadata, or dependency source when metadata is not
   available or when source-level checking is requested.
10. Resolve imports using the package graph and module index.
11. Lower AST to HIR with stable package/module/item identities.
12. Run name resolution, visibility checks, type checking, effect inference,
   handler checking, policy checking, limit checking, and determinism inference.
13. Validate declared entry points.
14. Validate reachable bodyless tool bindings for runnable targets.
15. Emit public package metadata, runnable artifacts, diagnostics, and index data.

The result of step 11 must not encode semantic identity as a filesystem path.
Resolved names use:

```text
PackageId + ModuleId + ItemId
```

### 13.2 Source Collection And Module Validation

The source root defaults to `src`:

```toml
[source]
root = "src"
```

Within that root, source files define logical modules:

```text
src/app/main.es      -> module app.main;
src/app/main/mod.es  -> module app.main;
src/app/main/cli.es  -> module app.main;
```

Rules:

- every package source file must declare exactly one `module`;
- the declared module path must match the path selected by the resolver;
- multiple source files may contribute to the same logical module;
- `src/app/main.es` and `src/app/main/mod.es` are competing canonical roots
  for `module app.main`, so using both in the same package is rejected;
- all parts of the same logical module share one top-level namespace, so
  duplicate top-level declarations in the merged module are rejected;
- imports are local to the source file that declares them; `public import` from
  any part contributes to the merged module's public API;
- relative filesystem imports are rejected;
- generated or virtual modules are allowed only when the package resolver maps
  them to stable logical module identities.

### 13.3 Dependency Compilation Modes

A dependency can be compiled in one of three modes:

| Mode | Input | Use case |
|---|---|---|
| Metadata mode | Generated public API metadata | Fast downstream type/effect/policy checking |
| Source mode | Dependency source files | Workspace development, debugging, local path dependencies |
| Built-in mode | Compiler-known metadata and optional source | Standard library and compiler-provided packages |

All modes must expose the same semantic surface to the current package:

- public modules and items;
- exported type, spec, flow, agent, tool, effect, action, and protocol
  signatures;
- generated effect/action summaries;
- tool schemas and binding requirements;
- re-export expansion;
- package identity and version.

### 13.4 Import Resolution During Compilation

Import resolution is project-level because an import path may refer to the
current package or to a dependency import root.

For:

```etas
import company.agents.Writer;
```

the compiler checks:

1. whether `company.agents.Writer` is a current-package module/item;
2. whether any dependency declares an import root that is a prefix of
   `company.agents.Writer`;
3. whether the longest matching prefix resolves to a public module or item;
4. whether the imported item is visible;
5. whether wildcard imports introduce ambiguity.

If multiple dependencies provide the same longest matching import root, the
import is ambiguous and compilation fails. The package manifest should use
different `import` roots or aliases to disambiguate.

### 13.5 Entry Point Validation

Runnable targets are declared in `etas.toml`:

```toml
[[bin]]
name = "research-assistant"
module = "app.main"
flow = "main"
```

The selected entry must resolve to a flow compatible with:

```etas
flow main(args: Array<string>) -> i32
```

The compiler may infer the return type and effect row of `main`, but the final
entry signature must have an `i32` return type. The inferred effect/action row of
the entry becomes the authority summary for deployment and runtime grant checks.

A package may define multiple runnable entries. Each entry is checked
independently, but they share the same package graph and module index.

### 13.6 Library Package Compilation

A package without `[[bin]]` entries can still be compiled as a library. Library
compilation validates the package and emits public package metadata. Public
exports must have stable signatures:

- public type and enum definitions;
- public flow, agent, and tool signatures;
- public effect and action declarations;
- public trace specs and protocol declarations;
- public re-exports after wildcard expansion;
- inferred or explicit effect/action summaries for public flows, agents, and
  tools.

The compiler should allow local/private flows to rely on inference, but public
metadata must record the resulting API contract for downstream packages.

### 13.7 Runnable Package Compilation

For a runnable target, compilation additionally checks runtime-facing
requirements:

- every reachable bodyless tool has a concrete binding;
- runtime-provided bindings declare effect/action rows compatible with source
  signatures;
- entry effects can be matched against available deployment grants;
- policies reachable from the entry are syntactically valid and can be checked
  against inferred actions;
- non-deterministic loops and retries have required `limit` clauses;
- handler scopes eliminate only the effects they claim to handle;
- generated runtime metadata includes source spans, stable item ids, entry
  effects, trace-spec summaries, limits, and tool schemas.

Compiling a runnable package does not execute it. Execution starts only when the
interpreter or runtime calls the selected entry flow.

### 13.8 Single-File And Editor Modes

Tools may support single-file parsing or checking for editor feedback:

```text
etas check file.es
```

Single-file mode is limited. It may parse syntax, infer local types, and report
local diagnostics, but full import resolution, dependency metadata loading,
entry validation, public API metadata generation, and tool binding validation
require project-level compilation.

## 14. Frontend Contract

The package manager may be implemented as a separate tool, but it must provide
the frontend with a resolved project input containing:

- the current package root and source root;
- package name, version, and edition;
- the selected entry module and flow;
- resolved dependency package identities;
- dependency import roots;
- dependency public API metadata;
- dependency effect/action metadata;
- dependency tool schemas and binding metadata when needed;
- lockfile fingerprints for reproducible checking and caching.

The frontend must not infer package identity from path strings alone. It should
lower every resolved reference to:

```text
PackageId + ModuleId + ItemId
```

## 15. Design Summary

| Concern | Location | Source syntax? |
|---|---|---|
| Module namespace | `.es` source via `module` | Yes |
| Importing names | `.es` source via `import` | Yes |
| Package name and version | `etas.toml` | No |
| Dependency source and version | `etas.toml` | No |
| Exact resolved dependency versions | `etas.lock` | No |
| Registry URLs and cache root | user/global config | No |
| Public API signatures and effects | generated package metadata | No |
| Tool host bindings | package metadata or runtime config | No |
| EDK official packages | ordinary dependencies such as `edk.http`, `edk.web`, and `edk.workspace` | No |

The source language stays small: `module`, `import`, visibility, and public
re-exports. Package management supplies the package graph and metadata that make
those imports reproducible, checkable, and analyzable.

EDK is the official package layer built on top of `std`, not a privileged source
language feature. Its package set and compatibility rules are described in
[EDK Official Packages](17-edk-official-packages.md).
